        Status do Azure DevOps
        O que cada status significa e o que fazer ao ver cada um.
        Fluxo normal: Proposed → Active → Awaiting QA → Resolved → Closed.
        Se der errado: Revised (volta corrigir) ou Refused (descartado). Paused = travado, ação imediata.
        Tempo Ideal por Status
        Quando é hora de agir — por tipo de projeto.

        Testes & Pareceres
        Como saber se os testes estão sendo feitos e como cobrar registros.
        Status de Teste
        StatusTraduçãoO que significa
        ✅ PassedAprovadoTestou e funcionou — pronto para Resolved
        ❌ FailedReprovadoTestou e encontrou erro — deve ir para Revised
        🚫 BlockedBloqueadoNão conseguiu nem testar — há impedimento
        ⏸️ Not RunNão executadoCriado mas nunca executado
        🔵 ActiveAtivoCriado, aguardando execução
        ⚠️ Armadilha mais comum
        Item Resolved ≠ item testado. Sempre cruze: item Resolved → tem Test Case com status Passed? Se não, não foi testado formalmente.
        Como verificar os testes
        • 1️⃣ Vá em Test Plans no DevOps
        • 2️⃣ Verifique se o item em Awaiting QA tem Test Case criado
        • 3️⃣ Veja se o Test Case foi executado (Passed, Failed ou Blocked)
        • 4️⃣ Test Case em "Not Run" há mais de 2 dias = teste não aconteceu
        • 5️⃣ Sempre pergunte: os testes foram em ambiente de homologação?
        Quando cobrar um Parecer
        • 📝 Resolved → Revised: QA deve ter registrado o motivo no Discussion
        • 📝 Item Paused: deve ter comentário explicando o bloqueio
        • 📝 Item Refused: motivo da recusa documentado
        • 📝 Teste Failed: descrição do erro no item
        Frase assertiva: "Por favor, registre o motivo diretamente no item do DevOps, na aba Discussion."
